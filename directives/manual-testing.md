# Directive: Manual Testing Checklist

## Goal
Generate interactive HTML test checklists from todo.md contracts so manual testing is structured, trackable, and feeds results back efficiently.

## When to Use
- A feature's final code step completes (all `[auto]` criteria pass) and unchecked `[manual]` items remain
- User runs `/snagging` to generate or regenerate a checklist
- A feature moves to `## Awaiting Sign-off` and needs a testing handoff
- Before merging a feature branch PR — snagging serves as the pre-merge verification gate

## Inputs
- `tasks/todo.md` — source of `[manual]` contract items per feature
- `STATE.md` — current app version and filename
- Optional: `--bugs` JSON file with known bugs found during automated code trace (generated by Claude during verification)

## Process

### Generation
1. Run `python3 execution/generate_test_checklist.py` (defaults to current feature from `## Current` in todo.md)
2. Script extracts all `[manual]` items, groups them by step, and generates `docs/{feature-slug}-manual-tests.html`
3. If automated testing found bugs (e.g. from `/agent-verify` code traces), pass them via `--bugs .tmp/test-bugs.json`
4. The checklist opens automatically in the browser

### Code verification before testing
Before presenting manual test steps to the user (or generating a checklist), verify function names, parameter types, valid input values, and API signatures against the actual code. Contracts are written at planning time and may use design-phase names that differ from the shipped implementation. Show exact working commands with real parameter values -- not design-phase abstractions from the contract. If a contract says "call X for any issue", check what X actually accepts and give the user a copy-pasteable command like `X('redistribution')`. This applies whether Claude is walking the user through tests verbally or generating a checklist HTML.

### Testing workflow
1. Open the generated checklist and the app side-by-side
1b. For visual verification, run `/chrome` to open the app in your browser for side-by-side testing
2. Work through each section in order — sections map to feature steps
3. Follow prerequisites at the top of each section (console commands have copy buttons)
4. Click each check item to cycle: **untested** → **pass** → **fail**
5. For failures: describe what you saw in the notes field that appears
6. Follow "Restore" callouts before moving to the next section
7. Progress persists in localStorage — you can close and resume later

### Feeding results back
1. Click **"Copy Test Results"** at the bottom of the checklist
2. Paste the results into the Claude conversation
3. Claude fixes failures and regenerates the checklist: `python3 execution/generate_test_checklist.py`
4. Re-test failed items only (previous passes are preserved in localStorage)
5. Repeat until all checks pass or failures are accepted as known issues

### Sign-off
- When all checks pass: Claude marks the `[manual]` items as `[x]` in todo.md. If on a feature branch, the PR can be merged. Feature moves to `## Done`
- When failures are accepted: document accepted issues in the feature's retro, mark `[manual]` items with a note, move to Done
- When failures need fixing: fix, regenerate checklist, re-test

## Outputs
- `docs/{feature-slug}-manual-tests.html` — interactive test checklist (version-controlled)
- Updated `[manual]` items in `tasks/todo.md` (marked [x] on sign-off)

## Edge Cases
- **No manual items**: Script exits with a message — nothing to generate. This is expected for `[INFRA]` features that are fully `[auto]`.
- **Feature not found**: Script exits with error listing available features.
- **Regeneration after fixes**: Previous test state is preserved in localStorage keyed by feature+version. If the version bumps, state resets (new version = new test cycle).
- **Multiple features in Awaiting Sign-off**: Run with `--feature "Feature Name"` to target a specific one.
- **Console commands in checks**: The checklist includes copy buttons for console commands. These are extracted from the test descriptions where they mention `localStorage` or console operations.

## Verification
- [ ] Generated HTML opens in browser and is interactive
- [ ] All `[manual]` items from todo.md appear as checks in the HTML
- [ ] Three-state toggle works (untested/pass/fail)
- [ ] Export button produces formatted test results
- [ ] localStorage persistence works across page reloads
