# Migration Manifest — kit v1.59.0

This manifest is consumed by `/pull-doe` pre-flight (see `directives/starter-kit-pull.md`). When a project pulls v1.59.0, the pre-flight phase greps the project for every `OLD:` phrase listed below and reports any matches as a "PULL IMPACT" warning. Behavioural changes (hook scope changes, new permission denies, etc.) are listed at the end with the grep patterns to detect at-risk project workflows.

**Pull impact summary.** v1.59.0 rewrites prohibition-style rules to positive-form instructions across the kit's directive surface. Rule meanings are preserved; only grammar changes. The two material behavioural changes are: (1) `permissions.deny` blocks `Read(.env)` at the tool layer, and (2) the `protect_directives` and `block_secrets_in_code` PreToolUse hooks gain coverage of the `bash` matcher (Tier 2 work — landing in Step 4).

## Format

Each phrase change uses three lines:

```
OLD: "<exact phrase removed from the file>"
NEW: "<phrase that replaced it>"
WHY: <one-line reason — usually positive form name, or load-bearing → decoration deletion>
```

Entries sit under `### <file path>` headings, grouped by tier. Behavioural changes use a different shape (see end). The manifest is consumed line-by-line by `/pull-doe`, so the OLD: / NEW: prefix matters for parsing.

## Behavioural changes

### `.claude/settings.json` — PreToolUse + PostToolUse matcher casing fix (Tier 2 work, lands Step 4)

Scope: PreToolUse and PostToolUse hook matchers across the kit's `.claude/settings.json`.
Old behaviour: matchers were declared in lowercase piped form (`write|create|edit`, `bash`, `read`) which does NOT match Claude Code's actual Tool name strings. Result: every PreToolUse and PostToolUse hook in the kit has been silently non-functional — the scripts were registered but never invoked. Confirmed empirically during Step 3 by observing that Anthropic's official `security-guidance` plugin (matcher `Edit|Write|MultiEdit`) DID fire while kit hooks (matcher `write|create|edit`) did NOT, despite manual stdin tests proving the kit hook scripts work when invoked.
New behaviour: matchers use case-sensitive Tool name strings — `Edit|Write|MultiEdit`, `Bash`, `Read`. After the fix, six PreToolUse hooks (`protect_directives`, `block_secrets_in_code`, `guard_kit_writes`, `block_dangerous_commands`, `enforce_review_gate`, `confirm_pr_merge`) and three PostToolUse hooks (`copy_plan_to_project`, `check_completed_feature`, `check_plan_freshness_hook`) start firing as designed.
Pull-impact for projects on v1.58.0 or earlier: their kit hooks have likewise been non-functional. After pulling v1.59.0, projects will newly observe the rules these hooks enforce: kit-write blocks (`/sync-doe` is now actually mandatory, not just documented), secret-detection blocks on Edit/Write, plan-copy automation, completed-feature warnings, plan-freshness reads. None of these change rule semantics — they enforce existing rules that were silently bypassed.
Pull-impact grep: `grep -E '"matcher": "(write|read|bash|edit)' .claude/settings.json` -- if a project has lowercase matchers, the v1.59.0 pull updates them. If a project has its own additional hooks with lowercase matchers, flag for manual review.

### `.claude/settings.json` — `permissions.deny` block

Scope: tool-layer permission (Read).
Old behaviour: nothing prevented Claude Code from invoking Read on `.env` or `.env.*` files; the existing `block_secrets_in_code.py` PreToolUse hook only gated writes.
New behaviour: Read is denied at the permission layer for `.env`, `./.env`, `./**/.env`, `.env.*`, and `./**/.env.*`. The model receives a permission-denied response before the file is opened.
Pull-impact grep: `grep -rnE 'Read\((\.env|.*/.env)' .` — if a project's docs or scripts reference reading .env via Claude tools, those references should be removed or updated.

### `protect_directives.py` — extended to bash matcher (Tier 2 work, lands Step 4)

Scope: PreToolUse, currently `write|create|edit` matcher only.
Old behaviour: writes via `Bash(cat > directives/X.md)`, `sed -i directives/X.md`, etc., bypass the hook.
New behaviour: hook also runs on the `bash` matcher and parses for redirected writes / in-place edits to `directives/`.
Pull-impact grep: `grep -rnE '(sed -i|cat *>|tee *>) .*directives/|>.* directives/' execution/ tasks/ .githooks/` — flag any project workflow that edits directives via Bash so it can be migrated to Edit/Write before pulling.

### `block_secrets_in_code.py` — extended to bash matcher (Tier 2 work, lands Step 4)

Scope: PreToolUse, currently `write|create|edit` matcher only.
Old behaviour: writes via `Bash(cat > path/with/secret …)` or `echo SECRET=… >> .env.local` bypass the hook.
New behaviour: hook also runs on the `bash` matcher and parses for redirected writes containing the secret pattern lexicon.
Pull-impact grep: `grep -rnE '(echo|printf|cat *>|tee).*-?key|-?token|-?secret' execution/ scripts/` — flag any project script that pipes secrets into files via Bash.

## Customised-directive check

Run from the project root before pulling:

```bash
PINNED_TAG=$(grep -oE 'kit v?[0-9]+\.[0-9]+\.[0-9]+' STATE.md | head -1 | grep -oE '[0-9]+\.[0-9]+\.[0-9]+')
KIT_DIR=~/doe-starter-kit
for f in directives/*.md; do
    diff -q "$f" "$KIT_DIR/$f" 2>/dev/null \
        || echo "PROJECT-CUSTOMISED (3-way merge needed): $f"
done
```

If any directive is project-customised, the v1.59.0 pull will require a manual 3-way merge for that file (project-edits + kit-edits-since-pin + kit v1.59.0 rewrite).
