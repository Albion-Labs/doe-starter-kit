## Tier 2 -- directives/

### directives/delivery-rules.md

OLD: - **Kit edits go through `/sync-doe`.** Versioning, tagging, and release happen automatically; the `guard_kit_writes` PreToolUse hook blocks direct writes.

NEW: - **Kit edits go through branches and PRs.** The kit's `.githooks/pre-commit` 'no direct-to-main' hook plus PR review are the canonical gate; the `guard_kit_writes` PreToolUse hook backs that up by blocking only irreversible Bash operations (recursive removal, force-push to kit main). For project-originated changes, `/sync-doe` is the translation tooling that strips project content, opens a kit branch, and opens the PR.

WHY: Reframes the kit-write rule around the actual canonical gate (.githooks/pre-commit + PR review). The PreToolUse hook becomes a defence-in-depth layer for irreversible operations only, not a global write block.

Pull-impact grep: `grep -rn 'Kit edits go through .sync-doe' .` -- locates the OLD phrasing in project mirrors. Replace with the NEW bullet.

## Tier 3 -- kit's CLAUDE.md + .claude/hooks/

### CLAUDE.md

WHY: Surfaces the new model at the top-level config that's loaded into every session.

### .claude/hooks/guard_kit_writes.py

BEHAVIOURAL CHANGE: file-edit branch (Edit/Write/MultiEdit) removed entirely; redirect/tee/cd-and-commit Bash patterns removed; rm-recursive of kit paths newly blocked; force-push to kit main (mentioning kit OR from kit cwd) newly blocked; FLAG_FILE bypass removed; cwd-aware logic kept only as a helper for force-push detection (no longer a top-level allow-early-return).

WHY: The OLD guard generated a long tail of false positives that made legitimate scripting work near-impossible from Claude Code. Empirical evidence (8 silent-guard releases v1.51 - v1.58, zero documented corruption) showed the OLD guard was protecting against a threat the AI fleet was not actually executing in practice. The new version provides STRICTLY STRONGER protection on irreversible operations (rm-r and force-push were either silently allowed or only partially caught in the OLD version), and weaker protection on accidental edits (mitigated by /sync-doe Step 0.5 dirty-tree pre-flight + PR review).

### .claude/hooks/protect_directives.py

BEHAVIOURAL CHANGE: `python3 -c "[..]directives[..]"` Bash pattern removed.

WHY: Pattern matched ANY Python one-liner referencing the directives/ token, including read-only (`os.listdir`), data-only (JSON payloads in test scaffolding), and string-literal mentions. False-positive rate during legitimate scripting was high enough to make `python3 -c` near-unusable from Claude Code (3 false-positive blocks during the v1.59.1 patch session alone). The narrow bypass route the pattern closed -- a Python one-liner that `open()`s a directive in write mode -- is theoretical; in practice writes go through Edit/Write/MultiEdit (file-path branch, still active and existence-aware) or Bash redirects (still covered by the >, >>, tee, sed, awk, rm, mv, cp patterns).

## Behavioural changes (less restrictive on edits, plus targeted strengthening)

1. `guard_kit_writes.py` -- file-edit branch retired. Edit/Write/MultiEdit on kit paths NOW ALLOWED at the PreToolUse layer.
   Pull-impact grep: `grep 'WRITE_TOOLS' .claude/hooks/guard_kit_writes.py` -- should return zero after pull.
   Mitigation: /sync-doe Step 0.5 dirty-tree pre-flight (new). PR review remains the canonical gate.

2. `guard_kit_writes.py` -- redirect, tee, cd-and-commit, cp, mv Bash patterns retired. `cat > kit/path`, `tee kit/path`, `cd kit && git commit`, `cp foo kit/x`, `mv foo kit/x` NOW ALLOWED.
   Pull-impact grep: `grep 'tee.*doe-starter-kit\|cd.*doe-starter-kit.*&&' .claude/hooks/guard_kit_writes.py` -- should return zero after pull.

3. `guard_kit_writes.py` -- rm-recursive of kit paths NEWLY BLOCKED. The OLD guard had no rm pattern at all; rm -rf of the kit working tree was silently allowed.
   Pull-impact grep: `grep 'rm.*recursive\|rm.*-r' .claude/hooks/guard_kit_writes.py` -- should match after pull.

4. `guard_kit_writes.py` -- force-push to kit (mentioning kit path OR from kit cwd) NEWLY BLOCKED. The OLD cd-and-commit pattern only caught force-push when prefixed with `cd kit &&`; from a shell already cd'd into kit (without `cd` in the command), it was silently allowed.
   Pull-impact grep: `grep 'git.*push.*force' .claude/hooks/guard_kit_writes.py` -- should match after pull.

5. `protect_directives.py` -- `python3 -c "[..]directives[..]"` Bash pattern retired.
   Pull-impact grep: `grep 'python3.*-c.*directives' .claude/hooks/protect_directives.py` -- should return zero after pull.

6. `.tmp/.sync-doe-active` flag file is no longer read by `guard_kit_writes.py`. `/sync-doe` may still touch and remove it for backwards compatibility with v1.59.x clients (harmless on v1.60.0+).
   Pull-impact grep: `grep 'FLAG_FILE\|sync-doe-active' .claude/hooks/guard_kit_writes.py` -- should return zero after pull.

## Customised-directive check

If your project has customised any of the affected directives or hooks, do a 3-way merge:
1. v1.59.x kit version (origin)
2. v1.60.0 kit version (new)
3. your project's current version (yours)

Most likely conflict points:
- `guard_kit_writes.py` -- if you customised the redirect/tee patterns or the FLAG_FILE bypass, those are gone in v1.60.0. Decide whether to retain a project-local override or accept the new model.
- `protect_directives.py` -- if you customised the python3 -c pattern, it is gone in v1.60.0. Decide similarly.
- `directives/delivery-rules.md` -- the kit-edit bullet is rewritten. Merge any project-specific addendum into the new bullet's context.
