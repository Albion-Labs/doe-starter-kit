# Migration Manifest — kit v1.62.0

**Pull impact summary.** v1.62.0 is a minor release that changes both phrasing and behaviour. The phrasing changes affect three universal templates that downstream projects may have copied verbatim (`universal-claude-md-template.md`, `SYSTEM-MAP.md`, `docs/tutorial/new-project.html`) — projects that adopted the pre-v1.60.0 "/sync-doe at feature end" framing in their own CLAUDE.md should update to the new "two ways to land a kit change, both PR-based" framing. The behavioural change is `/sync-doe` itself: it is now a conversational two-phase command (Phase 1 opens a PR, Phase 2 runs post-merge release machinery). The pre-flight greps below catch projects that mirror the old phrasings or that reference the retired `.tmp/.sync-doe-active` flag file.

## Tier 1 — universal templates + Core Behaviours

### universal-claude-md-template.md (kit's own — also the project-CLAUDE.md template)

OLD: "During feature work, make changes in the project repo first; run `/sync-doe` once at feature end to push them to the kit. The sync procedure handles tutorial footer stamping, changelog, tagging, pushing, AND creating the GitHub release. Direct commits to the kit skip all of these -- the tag may exist locally but no release appears on GitHub."

NEW: "**Kit changes flow through PRs** (v1.60.0+ PR-only model). The kit's `.githooks/pre-commit` 'no direct-to-main' hook plus human PR review are the canonical write gate. Two ways to land a kit change, both PR-based: (1) **Manual PR** for a one-off fix or improvement -- branch in the kit, commit, push, `gh pr create`, review, merge, same as any other repo. (2) **`/sync-doe`** for bundling many project-originated kit changes into a versioned release -- translates project-side improvements, opens a kit PR with the diff + CHANGELOG entry + version bump (Phase 1), then after merge runs the post-merge release machinery (Phase 2: tutorial stamp + tag + GitHub release) on main with a narrowly-scoped `SKIP_MAIN_PROTECTION=1` for the stamp commit only. Pick the manual PR for a one-line bug fix; pick `/sync-doe` when you have multiple project-originated changes ready to release."

WHY: the old phrasing implied `/sync-doe` was a release-bypass for direct-to-main commits. After v1.60.0's PR-only model + v1.62.0's two-phase `/sync-doe`, both paths to landing a kit change are PR-based. The new framing names that explicitly so the reader picks correctly.

### SYSTEM-MAP.md

OLD: "guard_kit_writes.py | `./.claude/hooks/` | Blocks direct writes to ~/doe-starter-kit (kit contributors only)"

NEW: "guard_kit_writes.py | `./.claude/hooks/` | Defence-in-depth: blocks irreversible Bash ops on kit paths (recursive removal, force-push to kit main). Not a file-edit gate -- PR review + the kit's `.githooks/pre-commit` 'no direct-to-main' hook are the canonical write gate (v1.60.0+)"

WHY: the v1.60.0 release retired `guard_kit_writes`'s file-edit branch; the hook now blocks only irreversible Bash patterns. The system-map description was factually outdated.

### docs/tutorial/new-project.html

OLD: "The kit itself stays in `~/doe-starter-kit/` and is never modified -- it copies files into your project."

NEW: "The wizard only copies files _from_ the kit _into_ your project; it never writes back to the kit."

WHY: scoped the reassurance to `setup.sh` (which IS read-only on the kit) instead of the absolute "never modified" — the kit IS modified via PRs by maintainers; just not by the wizard the new-project tutorial walks through.

## Behavioural changes

### `/sync-doe` is now a conversational two-phase command -- breaking change to the procedure

Scope: `directives/starter-kit-sync.md`, `global-commands/sync-doe.md`.

Old behaviour: `/sync-doe` translated project-side improvements, applied them to the kit working tree on `main`, bumped CHANGELOG, stamped tutorials, committed everything in one `chore(release): vX.Y.Z` commit (with `SKIP_MAIN_PROTECTION=1` covering the entire release content), tagged, pushed both tag and commit, and created the GitHub release. All atomic, all on `main`, no PR review.

New behaviour: `/sync-doe` is a **two-phase conversational command**.

- **Phase 1 (Step 10)** branches in the kit, commits the editorial content (translated diff + CHANGELOG entry + version bump in version files; **no tutorial stamp, no tag, no GitHub release** in Phase 1), pushes, opens a PR via `gh pr create`, writes a state file at `~/doe-starter-kit/.tmp/.sync-doe-pending-release.json` with `{version, prNumber, prUrl, branch, project, timestamp, phase}`, and stops. The maintainer reviews the PR and merges on GitHub.
- **Phase 2 (Step 11)** is gated on the PR being `MERGED` (verified via `gh pr view --json state`). Phase 2 pulls the merged main, runs the tutorial stamp + commit + tag + push (with `SKIP_MAIN_PROTECTION=1` narrowly scoped to the stamp commit only) + GitHub release. Idempotency checks (`git rev-parse "$VERSION"` + `gh release view "$VERSION"`) guard the stamp and release blocks so Phase 2 can be safely resumed after a partial failure.

Phase 2 routes are: explicit `/sync-doe` re-invocation (Step 0.7 detects the state file), or a conversational reply ("merged" / "yes" / "done") in any session.

Pull-impact for projects on v1.61.5: the next time you run `/sync-doe`, the procedure is fundamentally different. The slash command will pause after Phase 1 with "PR #N opened — reply 'merged' or re-run /sync-doe to release." This is intentional. **Read the new directive at `directives/starter-kit-sync.md` Steps 0.7, 10, 11 before your first post-pull `/sync-doe` invocation.**

Pull-impact grep: `! grep -rE 'directly on main|sync-doe-active|chore\(release\):.*sync from' --include='*.md' --include='*.html' . 2>/dev/null | grep -v 'migrations/\|CHANGELOG\.md'` — flags any project-side files that mirrored the pre-v1.62.0 phrasing or the retired flag file.

### `.tmp/.sync-doe-active` flag file is no longer touched by `/sync-doe`

Scope: `global-commands/sync-doe.md` (the slash command no longer instructs Claude to `touch` this file before kit writes).

Old behaviour: `/sync-doe` Step 11 said "Before writing ANY files to ~/doe-starter-kit, run `touch .tmp/.sync-doe-active`. This disables the kit write guard hook for this session."

New behaviour: the `touch` step is removed. The flag file was retired in v1.60.0 — the kit's `guard_kit_writes` hook no longer reads it (per `directives/kit-development.md` line 20). Carrying the touch step in the slash command was misleading users into thinking they were disabling an active hook. Phase 1 of the new flow doesn't need a kit-write bypass at all (it works through `git checkout -b` + `git commit` on a branch, which the v1.60.0 `guard_kit_writes` doesn't gate). Phase 2's stamp commit on `main` uses `SKIP_MAIN_PROTECTION=1` directly, no flag file needed.

Pull-impact for projects on v1.61.5: minor — projects that explicitly invoked `touch .tmp/.sync-doe-active` in their own scripts or hooks should remove those references. The flag file is still ignored by the v1.60.0+ hook, so leftover `touch` calls are no-ops, not regressions.

Pull-impact grep: `! grep -rn 'sync-doe-active' --include='*.md' --include='*.py' --include='*.sh' . 2>/dev/null | grep -v 'migrations/\|CHANGELOG\.md\|whats-new\.html'`

## Customised-directive check

Run from the project root before pulling:

```bash
PINNED_TAG=$(grep -oE 'kit v?[0-9]+\.[0-9]+\.[0-9]+' STATE.md | head -1 | grep -oE '[0-9]+\.[0-9]+\.[0-9]+')
KIT_DIR=~/doe-starter-kit
for f in directives/*.md; do
    diff -q "$f" "$KIT_DIR/$f" 2>/dev/null \
        || echo "PROJECT-CUSTOMISED (3-way merge needed): $f"
done
```
